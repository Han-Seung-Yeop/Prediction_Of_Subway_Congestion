# CROWDCAST · 최단 경로 안내 기능 기획서

> 작성일 2026-07-07 · 상태: **초안(리뷰 대기)** · 대상 브랜치: `main`

---

## 1. 개요

기존 CROWDCAST는 사용자가 `노선 → 역 → 방향 → 시간대`를 **직접** 골라야 해당 역의 칸별
혼잡도를 볼 수 있었다. 이번 기능은 여기에 **출발역 → 도착역 최단 경로 안내**를 추가하되,
단순 길찾기에 그치지 않고 **경로 자체를 혼잡도 예측의 입력으로 삼는다.**

### 핵심 컨셉 (한 줄 요약)

> **경로 = 입력, 칸별 혼잡도 = 출력.**
> 사용자는 출발·도착만 정하면, 앱이 경로를 계산하고 그 경로 위 역들에 진입하는 열차의
> **칸별 혼잡도**와 **추천 탑승/환승 칸**을 자동으로 보여준다.

기존의 수동 선택 방식(노선/역/방향/시간대)은 제거하지 않고 **탭/모드로 병행**한다.

## 2. 확정된 요구사항 (결정 로그)

| # | 항목 | 결정 |
|---|------|------|
| 1 | 경로 탐색 기준 | **최소 시간 · 최소 환승 · 최소 도보를 모두 종합 고려**한 추천 경로 |
| 2 | 혼잡도 시스템 역할 | 경로가 입력, 기존 칸별 혼잡도 3D/2D 시각화가 출력. 경로가 역·방향·시간을 대신 결정 |
| 3 | 혼잡도 표시 지점 | **탑승역·환승역 중심** + **하차/환승 시 최적 칸 추천**(동선 최적화) |
| 4 | 시간 기준 | **현재 시각 기본** + **출발 시각 선택** 둘 다 지원 |
| 5 | 경로 탐색 데이터 | **ODsay 대중교통 API** (지하철 경로·소요시간·환승) |
| 6 | 칸별 혼잡도 데이터 | **SK Open API — 진입역 기준 칸별 혼잡도** |
| 7 | API 연동 전략 | **목(mock) 응답으로 먼저 구현**, 실 API는 이후 교체 |
| 8 | 입력 UI | **간략 스키매틱 SVG 노선도**에서 역 클릭 선택 + **역 이름 검색(필수)** |
| 9 | 기존 수동 UI | **탭/모드로 병행** (경로 찾기 ↔ 역 직접 조회) |
| 10 | 교통 범위 | **지하철만** |
| 11 | 1차 결과물 | **기획서 문서 먼저** (본 문서) |

### "탭 병행이 앱을 무겁게 하지 않는가?" — 답

무겁지 않다. 경로 모드는 **입력 방식만 다를 뿐** 출력은 기존 칸별 혼잡도 렌더링
파이프라인(`predict.ts` → `Train3D`/`CarStrip`)을 그대로 재사용한다. 새로 붙는 것은
경로 상태·스키매틱 SVG·검색 UI·API 클라이언트(초기엔 목)뿐이며, three.js처럼 무거운
의존성이 추가되지 않는다. 스키매틱 SVG도 인라인 렌더라 번들 영향이 미미하다.

## 3. 사용자 플로우

### 3.1 경로 모드 (신규)

```
① 출발역 지정        → 스키매틱 노선도에서 역 클릭  또는  검색창 입력
② 도착역 지정        → 동일
③ (선택) 출발 시각    → 기본 '지금', 필요 시 시각 선택
④ [경로 찾기]        → ODsay(목) 호출 → 최소 시간·환승·도보 종합 경로 산출
⑤ 경로 요약 표시      → 총 소요시간 · 환승 횟수 · 도보 · 구간(leg) 목록
⑥ 구간 선택          → 각 탑승 구간의 '타는 역'을 SK(목)로 조회
⑦ 칸별 혼잡도 출력    → 기존 3D/2D 히트맵 + 추천 탑승 칸 + 하차/환승 최적 칸
```

### 3.2 역 직접 조회 모드 (기존, 유지)

현재 `App.tsx` 플로우를 탭 하나로 이동. 동작 변경 없음.

- **스키매틱 노선도**: 실제 지리 좌표가 아니라 직선·그리드형으로 단순화한 SVG.
  1차엔 기존 목업 4개 노선·20개 역을 대상으로 하고, 실 데이터 연동 시 역을 확장.
- **검색**: 역명 부분일치 자동완성. 노선도에 없는 역도 검색으로 선택 가능하게 설계
  (실 API 확장 대비).
- 기존 뷰 토글(3D/2D/역내부)과 `Legend`, `CarDetail`, `CarInterior3D`는 **그대로 재사용.**

## 5. 기능 명세

### 5.1 경로 탐색 (ODsay)

- 입력: 출발역·도착역 좌표(또는 역 ID), 출발 시각(현재/선택).
- 출력(정규화 후): 후보 경로 목록. 각 경로는 `총 소요시간`, `환승 횟수`, `총 도보`,
  `구간(leg) 배열`을 가진다.
- **종합 랭킹**: 최소 시간·최소 환승·최소 도보를 가중 합산한 점수로 대표 경로 1개를 추천하고,
  대안 경로도 함께 노출한다. 초기 가중치(조정 가능):

  ```
  score = w_time · 정규화(시간)
        + w_transfer · 정규화(환승수)
        + w_walk · 정규화(도보)
  // 초기값 예시: w_time 0.5, w_transfer 0.3, w_walk 0.2
  ```

  ODsay는 자체 정렬 결과도 주므로, 1차는 ODsay 순위를 신뢰하고 위 재정렬은 옵션으로 둔다.

### 5.2 칸별 혼잡도 매핑 (SK 진입역 기준 API)

- 각 **탑승 구간**마다 "타는 역(진입역)"과 방향이 정해진다 → 이 진입역 기준으로
  SK API에서 진입 열차의 칸별 혼잡도(%)를 받아 기존 히트맵으로 렌더.
- SK 실데이터가 없는 역/시간대는 **기존 `predict.ts` 추정 로직으로 폴백**한다
  (역 평균 × 칸별 가중치 + 이벤트 델타). → 실데이터/추정치 혼용, 출처 배지 표시.

### 5.3 추천 탑승·환승 칸 (동선 최적화)

- **추천 탑승 칸**: 해당 구간에서 가장 여유로운 칸(기존 `prediction.best`)을 기본 추천.
- **하차/환승 최적 칸**: 다음 액션(환승 or 최종 하차)의 이동거리를 최소화하는 칸.
  - 환승역의 `PlatformFeature` 중 `type: 'transfer'`(또는 최종 하차역의 `exit`) 위치 칸을
    "동선 최적 칸"으로 계산.
  - **혼잡도 vs 동선의 트레이드오프**를 함께 안내: "3번 칸이 환승 통로에 가장 가깝지만
    혼잡, 5번 칸은 한 칸 더 걷지만 여유" 식.
- 필요한 환승 통로 위치 데이터는 기존 `PLATFORM_FEATURES`를 확장(목업)하여 채운다.

### 5.4 시간 기준

- 기본은 **현재 시각**(`new Date().getHours()`).
- 사용자가 출발 시각을 바꾸면 그 시각 기준으로 재예측. 기존 `hour` state / `Controls`
  시간 슬라이더 로직을 경로 모드에서도 공유.

## 6. 데이터 · API 연동 설계

### 6.1 계층 분리 (핵심 원칙)

UI는 **어댑터 인터페이스**에만 의존하고, 그 뒤에서 목/실 API를 갈아끼운다.
이렇게 하면 "목 먼저 → 실 API 교체"가 UI 변경 없이 이뤄진다.

```
src/services/
  routing.ts      # interface RouteProvider  (findRoutes)
  routingMock.ts  # 목 데이터 구현
  routingOdsay.ts # ODsay 구현 (이후)
  congestion.ts   # interface CongestionProvider (carCongestionAt)
  congestionMock.ts  # predict.ts 재사용한 목
  congestionSk.ts    # SK Open API 구현 (이후)
```

### 6.2 인터페이스 초안 (타입 계약)

```ts
// routing.ts
export interface RouteLeg {
  lineId: string
  boardStationId: string      // 타는 역(진입역)
  alightStationId: string     // 내리는 역
  direction: 'up' | 'down'
  numStations: number
  rideMinutes: number
  isTransfer: boolean         // 이 leg 시작이 환승인지
}
export interface RoutePlan {
  legs: RouteLeg[]
  totalMinutes: number
  transferCount: number
  walkMinutes: number
  score: number               // 종합 랭킹 점수 (낮을수록 우선)
}
export interface RouteProvider {
  findRoutes(fromId: string, toId: string, departAt: Date): Promise<RoutePlan[]>
}

// congestion.ts
export interface CarCongestionResult {
  cars: CarCongestion[]       // 기존 타입 재사용
  source: 'sk-realtime' | 'estimate'   // 출처 배지용
  best: number
  worst: number
  transferOptimalCar?: number // 동선 최적 칸
}
export interface CongestionProvider {
  carCongestionAt(stationId: string, direction: 'up'|'down', at: Date): Promise<CarCongestionResult>
}
```

> 실제 ODsay/SK 요청·응답 필드명은 연동 시점에 각 API 문서 기준으로 확정한다.
> 위 인터페이스는 그 세부와 무관하게 UI를 고정하기 위한 **내부 정규화 형태**다.

### 6.3 API 키 · 보안 (연동 단계에서 처리)

- 1차(목)에서는 키 불필요.
- 실 API 연동 시 키 노출을 막기 위해 **경량 프록시**(Vite dev 서버 미들웨어 또는
  서버리스 함수)를 두는 방향으로 별도 결정. 프론트에서 직접 호출은 지양.

### 6.4 기존 코드 변경 지점

| 파일 | 변경 |
|------|------|
| `src/data/types.ts` | `RouteLeg`, `RoutePlan`, `CarCongestionResult` 등 타입 추가 |
| `src/data/subway.ts` | 노선/역 그래프 + 스키매틱 좌표(`x,y`) 필드 추가, 환승 통로 데이터 확장 |
| `src/data/predict.ts` | 그대로 재사용 (목 혼잡도 폴백 소스로 사용) |
| `src/services/*` | 신규 — 어댑터 계층 |
| `src/components/route/*` | 신규 — 검색, 스키매틱 노선도, 경로 요약, 구간 목록 |
| `src/App.tsx` | 모드 탭 추가, 경로 결과 → 기존 시각화에 주입 |

## 7. 단계별 구현 로드맵

> 각 단계는 독립적으로 동작 확인 가능하도록 쪼갬. **모두 목 데이터 기반.**

- **P0 — 뼈대**: 모드 탭 추가, 기존 화면을 "역 조회" 탭으로 이동. 회귀 없음 확인.
- **P1 — 입력 UI**: 역명 검색 + 스키매틱 SVG 노선도, 출발/도착/출발시각 상태.
- **P2 — 경로(목)**: `routingMock`으로 최소 시간·환승·도보 종합 경로 산출 + 경로 요약 UI.
- **P3 — 혼잡도 연결**: 선택 구간의 진입역 → `congestionMock`(=predict 재사용) → 기존
  3D/2D 히트맵에 주입. 출처 배지.
- **P4 — 추천 칸**: 추천 탑승 칸 + 하차/환승 최적 칸(동선) + 트레이드오프 안내.
- **P5 — 실 API 교체 준비**: ODsay/SK 어댑터 스텁 + 프록시/키 전략 결정 후 순차 교체.

## 8. 열린 이슈 (연동 단계에서 재확정)

1. **역 식별자 매핑**: 목업 `s-gangnam` ↔ ODsay/SK의 역 코드(외부 표준: 국토부/서울시
   지하철역 코드) 매핑 테이블 필요.
2. **랭킹 가중치**: `w_time/transfer/walk` 기본값과 사용자 조정 UI 노출 여부.
3. **SK 커버리지**: 실시간 칸 혼잡도가 없는 역/시간대 폴백 표기 방식(추정 배지 문구).
4. **프록시/키**: dev 미들웨어 vs 서버리스 함수 중 택1.
5. **스키매틱 노선도 확장**: 실 데이터 도입 시 전체 노선을 어떤 수준까지 그릴지.
